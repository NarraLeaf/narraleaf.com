The Host API is what a blueprint node's `execute` uses to drive a running game: open a page, move a widget, read a save, advance the dialog. It is the game-runtime counterpart to the editor's `app.services` ŌĆö those register things into the editor; the Host API acts on the live game.

## Where it lives

A node reaches the Host API from its execution context:

```ts
execute: async ctx => {
    const hostApi = ctx.hostAdapter.blueprintRuntime?.hostApi;
    if (!hostApi) return { nextPort: "next" }; // no game host ŌĆö nothing to do
    await hostApi.game.next();
    return { nextPort: "next" };
}
```

`hostApi` is present only in a **running game**: the Dev Mode window, Preview, and the exported Production build. The in-editor canvas preview omits it, so `ctx.hostAdapter.blueprintRuntime?.hostApi` is `undefined` there. Always guard for that ŌĆö a node that needs the game either skips its work or throws a clear error:

```ts
function getHostApi(ctx: Parameters<BlueprintNodeDef["execute"]>[0]) {
    const hostApi = ctx.hostAdapter.blueprintRuntime?.hostApi;
    if (!hostApi) {
        throw new Error("This node requires game host APIs");
    }
    return hostApi;
}
```

The same node `execute` runs in both places ŌĆö the studio entry registers it for the editor palette and preview, the runtime entry registers it for the game ŌĆö so the guard is the one thing that differs between environments. See [The two entries](/docs/studio/plugin/create-first-plugin).

## Two rules that hold everywhere

**Reads are pure, writes are effectful.** Every `get*` / `is*` is side-effect free and may run where the graph requires a pure value ŌĆö a Blueprint Value graph or a field binding. Every `set*` and every action (`next`, `openSurface`, `writeSave`, ŌĆ”) is effectful and cannot run in those places; it belongs on an execution flow.

**Values round-trip through JSON.** State, persistence, save metadata, frame data, and page props are serialized. Pass plain data (objects, arrays, strings, numbers, booleans, `null`); functions, class instances, and `undefined` do not survive. `undefined` is stored and returned as `null`.

## Versioning

The contract is versioned by `BLUEPRINT_HOST_API_CONTRACT_VERSION`, bumped only when the shape changes incompatibly. Nodes do not read it ŌĆö the host and the bundled types move together. It exists so a mismatch between a plugin built against an old contract and a newer host is detectable rather than silent.
